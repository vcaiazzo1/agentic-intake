The Problem

Manual triage doesn't scale.

⏱️

It's slow

Every request waits in a queue. A human reads it, decides where it goes, re-routes it. Average triage time: 8–15 minutes per ticket.

~12 min / ticket
🎲

It's inconsistent

Different agents classify the same request differently. Billing complaints land in the wrong queue. Critical bugs get treated as feature requests.

~20% mis-routing rate
📈

It doesn't scale

Doubling request volume means doubling headcount. Weekend spikes go unhandled until Monday. Escalation paths are manual and tribal.

Linear cost growth

The result: delayed responses, unhappy customers, and a support team spending most of its time on routing, not resolving.

The Coordinator

Structured classification, every time.

# Forced tool_choice — Claude MUST call this tool_choice = {"type": "tool", "name": "classify_request"} # Tool schema (excerpt) { "name": "classify_request", "input_schema": { "category": "bug_report | feature_request | billing_issue | general_question", "confidence": "float 0.0 – 1.0", "impact": "low | medium | high | critical", "reasoning": "step-by-step explanation" } } # Result → passed verbatim to specialist ClassificationResult( category = "bug_report", confidence = 0.92, impact = "high", reasoning = "Login failure after password reset..." )
  • 🔒
    Forced tool call
    tool_choice: {type: "tool"} — classification is always structured, never free-form prose.
  • 🎯
    Confidence scoring
    0.0–1.0 float. Below 0.6 triggers automatic escalation to a human.
  • ⚡
    Impact assessment
    critical impact always escalates, regardless of confidence.
  • 📋
    Reasoning passed downstream
    Specialist receives the coordinator's full reasoning so it never re-classifies.

The PreToolUse Hook

Audit trail for high-risk actions.

1
Intercept
Fires synchronously before generate_bug_ticket or flag_for_human execute. Unwatched tools pass through unaffected.
2
Classify Risk
flag_for_human → HIGH_RISK. generate_bug_ticket with severity=critical → CRITICAL, otherwise INFO.
3
Append to audit log
One JSONL line written to audit_log.jsonl — timestamp, tool, input, risk level, reason. Immutable append-only record.
4
Return allowed: true
Tool always executes. In production: check risk level, return allowed: false to block. The pattern, not the policy, is the demo.
audit_log.jsonl (sample entry)
{
"timestamp": "2026-04-28T09:14:22Z",
"tool": "generate_bug_ticket",
"input": {
"severity": "critical",
"component": "auth-service",
"title": "Login failure after reset"
},
"risk_level": "CRITICAL",
"reason": "critical severity bug ticket",
"allowed": true
}
append-only high-risk intercepted fully extensible

Most Inventive Claude Code Use

What we built — and how we built it.

01 / Agentic loops via prompt

Full multi-agent architecture from a single prompt

Coordinator, four specialists, escalation system, and pipeline wired together — each component generated in one focused Claude Code session with zero boilerplate from us.

"Build a coordinator.py that uses Claude's tool_use feature with a classify_request tool… Force the tool call with tool_choice…"
02 / adversarial eval design

Adversarial evaluation harness

We asked Claude Code to design the adversarial test cases and build the evaluation harness. It invented the ambiguous/misleading/edge-case taxonomy and wrote all 15 cases itself.

"Build an adversarial evaluation harness… compute strict accuracy, acceptable accuracy, false-confidence rate, and stratified breakdown by case type."
03 / hook as python function

PreToolUse hook wired into tool handlers

Rather than a separate Claude Code hook, we asked Claude Code to implement the pattern as a Python decorator inside the tool handler closures — no runtime magic, fully testable.

"Add a pre_tool_use_hook that intercepts generate_bug_ticket and flag_for_human calls and appends a JSONL entry to audit_log.jsonl… Return allowed=True always."
04 / offline test suite

Fully offline pytest suite via conftest injection

Claude Code injected a fake bedrock_client module into sys.modules in conftest.py before any project code imported — no AWS credentials needed, zero flaky tests.

"Mock the bedrock_client entirely in conftest.py by injecting a fake module into sys.modules before imports, so no AWS credentials are needed."
